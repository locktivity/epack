# Releasing epack

This document describes how to create a new release of epack.

## Prerequisites

- Push access to the repository

## Release Process

### 1. Update the Changelog

Edit `CHANGELOG.md` to move items from `[Unreleased]` to a new version section:

```markdown
## [Unreleased]

## [0.2.0] - 2025-03-15

### NEW FEATURES
- Add `--filter` flag to extract command

### BUG FIXES
- Fix digest mismatch on certain edge cases
```

### 2. Commit the Changelog

```bash
git add CHANGELOG.md
git commit -m "Release v0.2.0"
git push origin main
```

### 3. Create and Push the Tag

Tags must follow semver format with a `v` prefix:

```bash
git tag v0.2.0
git push origin v0.2.0
```

### 4. Automated Release

Pushing the tag triggers the release workflow (`.github/workflows/release.yaml`):

1. **Tests run** - All tests must pass
2. **SLSA Go Builder builds** - Creates binaries with SLSA Level 3 provenance:
   - `epack-{os}-{arch}` (full version: components)
   - `epack-core-{os}-{arch}` (minimal version: no components)
   - `*.intoto.jsonl` provenance attestations for each binary
3. **GitHub Release created** - With checksums and release notes
4. **Homebrew formulas updated** - In `locktivity/homebrew-tap`

### 5. Verify the Release

After the workflow completes:

1. Check the [GitHub Releases](https://github.com/locktivity/epack/releases) page
2. Verify all artifacts are present:
   - `epack-darwin-amd64`, `epack-darwin-arm64`
   - `epack-linux-amd64`, `epack-linux-arm64`
   - `epack-core-darwin-amd64`, `epack-core-darwin-arm64`
   - `epack-core-linux-amd64`, `epack-core-linux-arm64`
   - `*.intoto.jsonl` (provenance attestations)
   - `checksums.txt`
3. Test Homebrew installation:
   ```bash
   brew update
   brew install locktivity/tap/epack
   epack version
   ```
4. Verify SLSA provenance (optional):
   ```bash
   slsa-verifier verify-artifact epack-darwin-arm64 \
     --provenance-path epack-darwin-arm64.intoto.jsonl \
     --source-uri github.com/locktivity/epack \
     --source-tag v0.2.0
   ```

## SLSA Level 3 Provenance

All release binaries include SLSA Level 3 provenance attestations. This provides:

- **Non-forgeable provenance** - Generated by the [slsa-framework/slsa-github-generator](https://github.com/slsa-framework/slsa-github-generator) trusted builder
- **Isolated builds** - Build runs in a hardened, isolated environment
- **Source verification** - Provenance links binaries to the exact source commit

Users can verify provenance using [slsa-verifier](https://github.com/slsa-framework/slsa-verifier):

```bash
# Install verifier
go install github.com/slsa-framework/slsa-verifier/v2/cli/slsa-verifier@latest

# Verify binary
slsa-verifier verify-artifact epack-darwin-arm64 \
  --provenance-path epack-darwin-arm64.intoto.jsonl \
  --source-uri github.com/locktivity/epack
```

## Version Numbering

Follow [Semantic Versioning](https://semver.org/):

- **MAJOR** (1.0.0): Breaking changes to CLI flags, library API, or pack format
- **MINOR** (0.2.0): New features, backward-compatible
- **PATCH** (0.1.1): Bug fixes, backward-compatible

## Build Artifacts

The SLSA Go Builder produces two variants:

| Variant | Binary Pattern | Build Tags | Description |
|---------|----------------|------------|-------------|
| Full | `epack-{os}-{arch}` | `components` | All features: collector and tool orchestration |
| Core | `epack-core-{os}-{arch}` | (none) | Pack operations only, no components |

### Build Tags

epack uses Go build tags for optional features:

| Build Tag | Feature | Dependencies Added |
|-----------|---------|-------------------|
| `components` | Full component/remote/catalog/sdk surface (`new`, `init`, `lock`, `sync`, `install`, `update`, `collect`, `collector`, `tool`, `utility`, `catalog`, `push`, `pull`, `remote`, `sdk` commands) | (none beyond core) |

Build commands:
```bash
# Full: with components
go build -tags components ./cmd/epack

# Core: minimal dependencies
go build ./cmd/epack
```

Both variants embed version metadata via ldflags:
- `Version` - Git tag (e.g., `0.2.0`)
- `Commit` - Git commit SHA
- `BuildDate` - Build timestamp

Build configurations are in `.slsa-goreleaser/`.

## Troubleshooting

### Release workflow failed

1. Check the [Actions tab](https://github.com/locktivity/epack/actions) for error details
2. Common issues:
   - Tests failing - fix and re-tag
   - `HOMEBREW_TAP_TOKEN` expired - regenerate the PAT
   - SLSA builder version mismatch - update `@v2.1.0` reference if needed

### Re-releasing a version

If you need to re-release (e.g., workflow failed after partial completion):

```bash
# Delete the remote tag
git push origin :refs/tags/v0.2.0

# Delete the local tag
git tag -d v0.2.0

# Fix the issue, then re-tag
git tag v0.2.0
git push origin v0.2.0
```

**Note:** If the GitHub Release was already created, delete it manually before re-pushing the tag.

### Homebrew formula not updated

The `HOMEBREW_TAP_TOKEN` must have `repo` scope for `locktivity/homebrew-tap`. Verify:
1. The token hasn't expired
2. The token has correct permissions
3. The `locktivity/homebrew-tap` repository exists with `Formula/` directory

### SLSA provenance verification fails

1. Ensure you're using the correct provenance file (matches the binary)
2. Check that the source URI matches: `github.com/locktivity/epack`
3. For tag verification, use `--source-tag vX.Y.Z`

## Manual Release (Emergency)

If automated release fails completely, you can build locally but **this will not produce SLSA Level 3 provenance**:

```bash
# Build locally (no provenance)
CGO_ENABLED=0 go build -tags components -o epack-darwin-arm64 ./cmd/epack
CGO_ENABLED=0 go build -o epack-core-darwin-arm64 ./cmd/epack

# Upload artifacts manually to GitHub Releases
# Update Homebrew formulas manually
```

For true SLSA Level 3 builds, the release must go through the GitHub Actions workflow.
